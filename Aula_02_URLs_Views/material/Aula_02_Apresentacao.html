📦 O que são Apps Django?

Projeto vs App

Projeto:

  • Configuração geral
  • Settings centralizados
  • URLs principais

App:

  • Funcionalidade específica
  • Blog, Loja, Fórum...
  • Reutilizável

✅ Vantagens

  • Organização modular
  • Reutilização em outros projetos
  • Separação de responsabilidades
  • Manutenção facilitada
  • Trabalho em equipe

📝 Registrando o App

Abra settings.py e adicione em INSTALLED_APPS:


INSTALLED_APPS = [

    'django.contrib.contenttypes',

💡 Por que registrar?
O Django precisa saber quais apps fazem parte do projeto para carregar templates, static files, migrations, etc.

🔀 Como funciona o Roteamento

🌐 Requisição: http://site.com/blog/5/
↓
📄 urls.py (projeto) - ROOT_URLCONF
↓
📄 urls.py (app) - include()
↓
⚙️ View Function ou Class-Based
↓
📤 HttpResponse (HTML, JSON, etc)

🗂️ URLs no Projeto Principal

meu_projeto/urls.py:


from django.contrib import admin
from django.urls import path, include

urlpatterns = [
    # Admin
    path('admin/', admin.site.urls),
    
    # Apps
    path('', include('home.urls')),           # Página inicial
    path('blog/', include('blog.urls')),      # Blog
    path('contato/', include('contato.urls')),# Contato
]
💡 O include() delega o roteamento para o app específico

📋 URLs dentro de Apps

Crie blog/urls.py:


from django.urls import path
from . import views

app_name = 'blog'  # Namespace do app

urlpatterns = [
    path('', views.lista_posts, name='lista'),
    path('<int:post_id>/', views.detalhe_post, name='detalhe'),
    path('novo/', views.criar_post, name='criar'),
    path('<int:post_id>/editar/', views.editar_post, name='editar'),
    path('categoria/<slug:slug>/', views.posts_por_categoria, 
         name='categoria'),
]

🔧 Função path()


path(route, view, kwargs=None, name=None)
Parâmetros:
  1. route: String com o padrão da URL
  2. view: Função ou classe de view
  3. kwargs: Argumentos extras (opcional)
  4. name: Nome único para a URL
💡 Dica:
Sempre use o parâmetro name! Facilita referências em templates e código.

🔄 Path Converters

Converter Descrição Exemplo Uso
str Qualquer string (exceto /) <str:nome> Nomes de usuário
int Inteiro positivo ou zero <int:id> IDs numéricos
slug Slug (letras, números, - e _) <slug:slug> URLs amigáveis
uuid UUID format <uuid:id> IDs únicos
path Qualquer string (inclui /) <path:caminho> Caminhos completos

💡 Exemplos Práticos


urlpatterns = [
    # str: captura qualquer texto
    path('perfil/<str:username>/', views.perfil),
    # Captura: /perfil/joao123/
    
    # int: apenas números
    path('produto/<int:produto_id>/', views.produto),
    # Captura: /produto/42/
    # Não captura: /produto/abc/
    
    # slug: formato de URL amigável
    path('artigo/<slug:slug>/', views.artigo),
    # Captura: /artigo/django-para-iniciantes/
    
    # múltiplos parâmetros
    path('categoria/<slug:categoria>/<int:ano>/', views.por_ano),
]

👁️ O que são Views?

Views são funções ou classes Python que:
  • Recebem requisições HTTP (HttpRequest)
  • Processam dados e lógica de negócio
  • Retornam respostas HTTP (HttpResponse)
Function-Based Views (FBV)

Simples, diretas, ideais para iniciantes

Class-Based Views (CBV)

Reutilizáveis, herança, mais complexas

🎨 Sua Primeira View

blog/views.py:


from django.http import HttpResponse

def index(request):
    """
    View simples que retorna texto
    """
    return HttpResponse("Bem-vindo ao Blog!")

def sobre(request):
    """
    View com HTML inline
    """
    html = """
    <html>
        <body>
            <h1>Sobre o Blog</h1>
            <p>Este é um blog sobre Django.</p>
        </body>
    </html>
    """
    return HttpResponse(html)

📊 View com Parâmetros


from django.http import HttpResponse

def detalhe_post(request, post_id):
    """
    View que recebe parâmetro da URL
    """
    return HttpResponse(f"Você está vendo o post #{post_id}")

def posts_por_ano(request, ano, mes):
    """
    View com múltiplos parâmetros
    """
    return HttpResponse(f"Posts de {mes}/{ano}")

def perfil(request, username):
    """
    View com parâmetro de texto
    """
    return HttpResponse(f"Perfil de {username}")

📨 Objeto HttpRequest

O Django cria automaticamente um objeto HttpRequest com informações da requisição:

Atributos importantes:
  • request.method - GET, POST...
  • request.GET - Query params
  • request.POST - Form data
  • request.FILES - Arquivos
  • request.user - Usuário
Métodos úteis:
  • request.is_ajax()
  • request.get_host()
  • request.get_full_path()
  • request.META - Headers

📤 Objeto HttpResponse


from django.http import (
    HttpResponse, 
    JsonResponse, 
    HttpResponseRedirect,
    HttpResponseNotFound,
    HttpResponseForbidden
)

# Resposta simples
def view1(request):
    return HttpResponse("Olá!")

# Resposta JSON
def view2(request):
    data = {'nome': 'João', 'idade': 25}
    return JsonResponse(data)

# Redirecionamento
def view3(request):
    return HttpResponseRedirect('/outra-url/')

# Erro 404
def view4(request):
    return HttpResponseNotFound("Página não encontrada")

🔄 Métodos GET e POST

GET

  • Buscar/ler dados
  • Parâmetros na URL
  • Pode ser cacheado
  • Idempotente

# /busca/?q=django&pg=1
def busca(request):
    termo = request.GET.get('q')
    pagina = request.GET.get('pg', 1)
    return HttpResponse(
        f"Buscando: {termo}"
    )

POST

  • Criar/modificar dados
  • Dados no corpo
  • Não cacheado
  • Não idempotente

def criar_post(request):
    if request.method == 'POST':
        titulo = request.POST.get('titulo')
        conteudo = request.POST.get('conteudo')
        # Salvar no banco
        return HttpResponse("Post criado!")
    return HttpResponse("Formulário")

🔒 CSRF Protection

CSRF (Cross-Site Request Forgery): Ataque que força um usuário autenticado a executar ações indesejadas.

Django protege automaticamente com tokens CSRF:


<form method="POST">
    {% csrf_token %}
    <input type="text" name="titulo">
    <button type="submit">Enviar</button>
</form>
⚠️ Importante: Sempre use {% csrf_token %} em formulários POST!

⚡ Shortcuts Úteis


from django.shortcuts import render, redirect, get_object_or_404

# Renderizar template
def index(request):
    return render(request, 'blog/index.html', {'titulo': 'Blog'})

# Redirecionamento
def logout(request):
    # Lógica de logout
    return redirect('blog:index')  # Redireciona para URL nomeada

# Buscar ou retornar 404
def detalhe(request, post_id):
    post = get_object_or_404(Post, pk=post_id)
    return render(request, 'blog/detalhe.html', {'post': post})

🏷️ URLs Nomeadas

Em urls.py


# blog/urls.py
app_name = 'blog'

urlpatterns = [
    path('', views.index, 
         name='index'),
    path('<int:id>/', 
         views.detalhe, 
         name='detalhe'),
]

Em templates


<!-- Referência estática -->
<a href="/blog/">Blog</a>

<!-- URL nomeada (recomendado) -->
<a href="{% url 'blog:index' %}">
    Blog
</a>

<a href="{% url 'blog:detalhe' post.id %}">
    Ver Post
</a>

📝 Exemplo Completo


# blog/views.py
from django.shortcuts import render
from django.http import HttpResponse

def lista_posts(request):
    """Lista todos os posts"""
    posts = [
        {'id': 1, 'titulo': 'Primeiro Post', 'autor': 'João'},
        {'id': 2, 'titulo': 'Segundo Post', 'autor': 'Maria'},
        {'id': 3, 'titulo': 'Terceiro Post', 'autor': 'Pedro'},
    ]
    context = {
        'posts': posts,
        'total': len(posts)
    }
    return render(request, 'blog/lista.html', context)

def detalhe_post(request, post_id):
    """Mostra detalhes de um post"""
    # Simulando busca no banco
    post = {'id': post_id, 'titulo': f'Post #{post_id}'}
    return render(request, 'blog/detalhe.html', {'post': post})

✅ Boas Práticas

  • Use URLs nomeadas sempre que possível
  • Organize apps por funcionalidade
  • Mantenha views simples e focadas
  • Use shortcuts do Django (render, redirect)
  • Sempre inclua {% csrf_token %} em forms POST
  • Valide dados recebidos de formulários
  • Use namespaces para evitar conflitos de URLs
  • Documente suas views com docstrings

⚠️ Erros Comuns

❌ Esquecer de registrar o app
Sempre adicione o app em INSTALLED_APPS
❌ URLs sem nome
Dificulta manutenção. Use sempre name='...'
❌ Esquecer csrf_token
Forms POST precisam de {% csrf_token %}
❌ Confundir ordem das URLs
URLs mais específicas devem vir antes das genéricas
