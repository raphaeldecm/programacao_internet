Ambiente Virtual Python Setup

Isolando as dependências do projeto

Um ambiente virtual (virtualenv) cria um espaço isolado para as dependências Python do projeto, evitando conflitos com outros projetos ou com a instalação global do Python. É uma boa prática indispensável antes de qualquer projeto Django.

Passo 1 — Criar o diretório e o ambiente virtual
Terminal
# Cria o diretório do projeto $ mkdir mercado && cd mercado # Cria o ambiente virtual na pasta 'venv' $ python -m venv venv
Passo 2 — Ativar o ambiente virtual
Terminal
# Linux / macOS $ source venv/bin/activate # Windows (PowerShell) PS> venv\Scripts\Activate.ps1 # Windows (cmd.exe) > venv\Scripts\activate.bat
Após a ativação, o prompt exibirá (venv) no início da linha, indicando que o ambiente está ativo. Todas as instalações feitas com pip afetarão apenas este ambiente.
O diretório venv/ não deve ser incluído no repositório Git. Adicione-o ao .gitignore: echo "venv/" >> .gitignore
04

Instalação do Django Setup

Instalando e verificando a versão

Com o ambiente virtual ativo, utilizamos o pip para instalar o Django. O pip buscará o pacote no PyPI e instalará o Django junto com suas dependências dentro do ambiente virtual.

Instalar o Django
Terminal — com (venv) ativo
(venv) $ pip install django
Verificar a versão instalada
Terminal
(venv) $ python -m django --version # Saída esperada: 5.x.x (ou a versão instalada)
Gerar arquivo de dependências
Terminal
(venv) $ pip freeze > requirements.txt # Registra todas as dependências instaladas
O arquivo requirements.txt permite que qualquer outro desenvolvedor recrie o ambiente com pip install -r requirements.txt.
05

Criando o Projeto Django Setup

O comando django-admin startproject

O comando django-admin startproject gera a estrutura base de um projeto Django: o arquivo de configurações, o roteador principal de URLs, as configurações de WSGI/ASGI e o manage.py.

Criar o projeto mercado
Terminal
# O ponto (.) cria o projeto no diretório atual, sem subpasta extra (venv) $ django-admin startproject mercado .
O uso do ponto . no final é uma convenção importante: evita a criação de um subdiretório mercado/mercado/ redundante, mantendo o manage.py na raiz do projeto.
Testar o servidor de desenvolvimento
Terminal
(venv) $ python manage.py runserver # Acesse: http://127.0.0.1:8000 # Você verá a página padrão "The install worked successfully!"
O aviso sobre migrações pendentes (unapplied migrations) é normal neste momento. Pode ser ignorado por enquanto; ele indica que as tabelas do banco ainda não foram criadas.

Estrutura Inicial do Projeto Setup

Entendendo cada arquivo gerado
Árvore de diretórios após startproject
mercado/ ← raiz do projeto (diretório de trabalho) ├── mercado/ ← pacote de configuração do projeto │ ├── __init__.py ← marca como pacote Python │ ├── settings.py ← todas as configurações do projeto │ ├── urls.py ← roteador principal de URLs │ ├── asgi.py ← ponto de entrada para servidores ASGI │ └── wsgi.py ← ponto de entrada para servidores WSGI ├── manage.py ← utilitário de linha de comando do Django ├── requirements.txt← dependências do projeto └── venv/ ← ambiente virtual (não vai ao Git)
manage.py

Ponto de entrada para comandos como runserver, migrate, startapp, shell, entre outros. Nunca deve ser editado manualmente.

settings.py

Centraliza todas as configurações: banco de dados, apps instalados, templates, arquivos estáticos, chave secreta, debug e muito mais.

urls.py

Define o roteamento principal do projeto. As URLs dos apps são incluídas aqui com include(), mantendo a organização modular.

wsgi.py / asgi.py

Interfaces de comunicação entre o Django e o servidor web em produção. Raramente são editados durante o desenvolvimento.

Criando o App Produtos Setup

O comando manage.py startapp

Um app no Django é um módulo Python autossuficiente que encapsula uma funcionalidade específica. O projeto mercado pode ter vários apps: produtos, clientes, pedidos, etc. Cada app pode ser reutilizado em outros projetos.

Criar o app produtos
Terminal
(venv) $ python manage.py startapp produtos
Estrutura gerada pelo startapp
produtos/ ├── __init__.py ← marca como pacote Python ├── admin.py ← registro de models no Django Admin ├── apps.py ← configuração do app (nome, label) ├── models.py ← definição dos models (tabelas do BD) ├── tests.py ← testes automatizados do app ├── views.py ← lógica das views (funções ou classes) └── migrations/ ← arquivos de migração do banco de dados └── __init__.py
Criar o app com startapp não é suficiente — é necessário registrá-lo no settings.py. Isso será feito no próximo slide.
08

Configurando o settings.py Config

Registrando o app em INSTALLED_APPS

O Django precisa conhecer os apps que fazem parte do projeto. Isso é feito adicionando o app à lista INSTALLED_APPS no settings.py. A string adicionada referencia o arquivo apps.py do app.

mercado/settings.py — seção INSTALLED_APPS editar
⚙️ mercado/settings.py
INSTALLED_APPS = [ 'django.contrib.admin', 'django.contrib.auth', 'django.contrib.contenttypes', 'django.contrib.sessions', 'django.contrib.messages', 'django.contrib.staticfiles', # Apps do projeto 'produtos.apps.ProdutosConfig', # ← nosso app ]
A string 'produtos.apps.ProdutosConfig' aponta para a classe ProdutosConfig dentro do arquivo produtos/apps.py. Essa é a forma recomendada, pois permite que o Django carregue metadados do app corretamente.
produtos/apps.py — gerado automaticamente
🐍 produtos/apps.py
from django.apps import AppConfig class ProdutosConfig(AppConfig): default_auto_field = 'django.db.models.BigAutoField' name = 'produtos'
09

Configurando Templates Config

Definindo onde o Django buscará os arquivos HTML

O Django localiza arquivos de template com base na configuração TEMPLATES no settings.py. Por padrão, a chave DIRS está vazia — é preciso apontar para o diretório global de templates do projeto. Além disso, APP_DIRS: True instrui o Django a buscar templates dentro de cada app, na subpasta templates/.

mercado/settings.py — seção TEMPLATES editar
⚙️ mercado/settings.py
from pathlib import Path BASE_DIR = Path(__file__).resolve().parent.parent TEMPLATES = [ { 'BACKEND': 'django.template.backends.django.DjangoTemplates', 'DIRS': [BASE_DIR / 'templates'], # ← pasta global de templates 'APP_DIRS': True, # ← busca em app/templates/ automaticamente 'OPTIONS': { 'context_processors': [ 'django.template.context_processors.debug', 'django.template.context_processors.request', 'django.contrib.auth.context_processors.auth', 'django.contrib.messages.context_processors.messages', ], }, }, ]
DIRS define caminhos explícitos onde o Django buscará templates. Aqui aponta para mercado/templates/ — a pasta global do projeto. Use BASE_DIR / 'templates' para construir o caminho de forma portável.
APP_DIRS: True habilita a busca automática em nome_do_app/templates/ de cada app instalado. Quando ativado, não é necessário listar esses caminhos manualmente em DIRS.
10

Arquivos Estáticos — Conceitos Static

CSS, JavaScript e imagens no Django

Arquivos estáticos são recursos que não mudam dinamicamente: folhas de estilo CSS, scripts JavaScript, imagens, fontes, etc. O Django os gerencia por meio do app django.contrib.staticfiles, que já vem habilitado por padrão.

As três configurações essenciais
  • 1
    STATIC_URL — prefixo da URL pela qual os arquivos serão acessados no navegador. Ex.: /static/ significa que style.css será acessado em /static/style.css.
  • 2
    STATICFILES_DIRS — lista de diretórios onde o Django buscará arquivos estáticos além dos diretórios static/ internos de cada app. Não existe por padrão no settings.py — precisa ser criado manualmente.
  • 3
    STATIC_ROOT — diretório destino usado pelo comando collectstatic em produção, para centralizar todos os arquivos estáticos. Não é necessário em desenvolvimento.
Em desenvolvimento (com DEBUG=True), o Django serve os estáticos automaticamente. Em produção, o servidor web (Nginx, Apache) deve ser configurado para servir a pasta STATIC_ROOT.
11

STATIC_URL e STATICFILES_DIRS Static

Configurando os caminhos de arquivos estáticos

Ao final do settings.py, localize a variável STATIC_URL (já existe) e adicione logo abaixo a variável STATICFILES_DIRS, que não existe por padrão e deve ser criada manualmente.

mercado/settings.py — seção de estáticos editar
⚙️ mercado/settings.py
# Já existe — apenas confirme o valor STATIC_URL = 'static/' # NÃO existe por padrão — deve ser CRIADA manualmente STATICFILES_DIRS = [ BASE_DIR / 'static', # ← pasta global de estáticos do projeto ] # Necessário apenas em produção (collectstatic) # STATIC_ROOT = BASE_DIR / 'staticfiles'
STATICFILES_DIRS não vem definido no settings.py padrão do Django. É necessário criá-la explicitamente. Ela aponta para diretórios globais de estáticos — separados dos diretórios static/ internos de cada app.
A pasta static/ na raiz do projeto (referenciada por STATICFILES_DIRS) é destinada a arquivos compartilhados entre todos os apps — como um CSS base ou logo do projeto. Cada app pode ter seu próprio diretório static/ interno.

Estrutura de Diretórios Final Config

Como o projeto ficará organizado
mercado/ ← raiz do projeto ├── mercado/ ← pacote de configuração │ ├── __init__.py │ ├── settings.py │ ├── urls.py │ └── wsgi.py / asgi.py │ ├── produtos/ ← app do projeto │ ├── static/ ← estáticos do app (APP_DIRS) │ │ └── produtos/ ← namespace: evita colisão │ │ ├── style.css │ │ ├── script.js │ │ └── img/ │ ├── templates/ ← templates do app (APP_DIRS) │ │ └── produtos/ ← namespace: evita colisão │ │ └── lista.html │ ├── views.py │ └── urls.py │ ├── static/ ← estáticos globais (STATICFILES_DIRS) │ ├── base.css │ └── img/ │ ├── templates/ ← templates globais (DIRS) │ └── home.html │ ├── manage.py └── requirements.txt
Note o padrão de subpasta com o nome do app dentro de templates/ e static/ do app (produtos/templates/produtos/). Isso cria um namespace de arquivos e evita conflitos quando dois apps têm um arquivo com o mesmo nome.
13

URL Conf do Projeto Principal URLs

Roteamento central com include()

O arquivo mercado/urls.py é o roteador principal do projeto. Ele define a URL home do projeto e usa include() para delegar as URLs do app produtos ao arquivo urls.py do próprio app.

mercado/urls.py editar
🔗 mercado/urls.py
from django.contrib import admin from django.urls import path, include from django.shortcuts import render # View simples da home — definida aqui por conveniência def home(request): return render(request, 'home.html') urlpatterns = [ path('admin/', admin.site.urls), # URL da página inicial do projeto path('', home, name='home'), # Delega as URLs do app produtos path('produtos/', include('produtos.urls')), ]
path('', home, name='home') mapeia a URL raiz (/) para a função home. O parâmetro name permite referenciar a URL pelo nome em templates e views.
include('produtos.urls') delega qualquer URL que comece com /produtos/ para o arquivo urls.py do app, mantendo o código organizado e modular.
14

View da Home View

Renderizando o template home.html

A função home é uma view baseada em função (FBV — Function-Based View). Ela recebe o objeto request (representando a requisição HTTP) e retorna um objeto HttpResponse — neste caso, gerado pelo atalho render(), que processa o template e devolve o HTML resultante.

A função render() — entendendo os parâmetros
🐍 mercado/urls.py (trecho)
from django.shortcuts import render def home(request): # render(request, template_name, context=None) # ├── request → objeto da requisição HTTP # ├── 'home.html' → caminho do template (relativo a DIRS ou app/templates) # └── context → dicionário com dados para o template (opcional) return render(request, 'home.html')
templates/home.html novo
🌐 templates/home.html
<!DOCTYPE html> <html lang="pt-br"> <head> <meta charset="UTF-8"> <title>Mercado — Home</title> </head> <body> <h1>Bem-vindo ao Mercado</h1> </body> </html>
O Django buscará home.html primeiro nos caminhos definidos em DIRS (a pasta templates/ na raiz do projeto) e depois nas pastas templates/ de cada app instalado, graças a APP_DIRS: True.

URLs do App Produtos URLs

Criando o urls.py dentro do app

O app produtos não possui um urls.py gerado automaticamente pelo startapp. Precisamos criar esse arquivo manualmente e nele definir as rotas específicas do app.

produtos/urls.py criar
🔗 produtos/urls.py
from django.urls import path from . import views urlpatterns = [ # /produtos/ → lista de produtos path('', views.lista, name='lista'), # /produtos/detalhes/ → detalhe de produto path('detalhes/', views.detalhes, name='detalhes'), ]
produtos/views.py editar
🐍 produtos/views.py
from django.shortcuts import render def lista(request): return render(request, 'produtos/lista.html') def detalhes(request): return render(request, 'produtos/detalhes.html')
Note que o caminho do template é 'produtos/lista.html' — incluindo o nome do app como prefixo. Isso corresponde ao arquivo em produtos/templates/produtos/lista.html, seguindo a convenção de namespace por subpasta.
16

Namespace nas URLs URLs

Evitando conflitos de nomes entre apps

Quando um projeto tem vários apps, é comum que URLs com o mesmo name colidam. O namespace resolve isso: permite referenciar URLs como produtos:lista em vez de apenas lista, tornando o código mais claro e sem ambiguidade.

Definir app_name no urls.py do app editar
🔗 produtos/urls.py
from django.urls import path from . import views # Define o namespace do app app_name = 'produtos' urlpatterns = [ path('', views.lista, name='lista'), path('detalhes/', views.detalhes, name='detalhes'), ]
Uso do namespace em templates HTML
🌐 qualquer template
<!-- Sem namespace (ambíguo se houver outros apps com 'lista') --> <a href="{% url 'lista' %}">Produtos</a> <!-- Com namespace (claro e sem colisão) --> <a href="{% url 'produtos:lista' %}">Produtos</a> <a href="{% url 'produtos:detalhes' %}">Detalhes</a>
A sintaxe {% url 'namespace:nome' %} é a template tag do Django para gerar URLs dinamicamente. Ela garante que a URL seja atualizada automaticamente caso a rota mude no urls.py, evitando URLs hardcoded nos templates.

Templates do Projeto Templates

Arquivos HTML na pasta global templates/

A pasta templates/ na raiz do projeto armazena templates de alcance global, como a página inicial (home.html) e eventuais templates base (layouts compartilhados). Esses arquivos são acessíveis a partir de qualquer view do projeto.

templates/home.html — com link para o app editar
🌐 templates/home.html
<!DOCTYPE html> <html lang="pt-br"> <head> <meta charset="UTF-8"> <title>Mercado — Home</title> </head> <body> <h1>Bem-vindo ao Mercado</h1> <p>Explore nossa loja.</p> <!-- Uso do namespace para o link do app --> <a href="{% url 'produtos:lista' %}">Ver Produtos</a> </body> </html>
A tag {% url 'produtos:lista' %} é resolvida em tempo de renderização pelo Django, gerando a URL correspondente à view lista no app produtos. Isso é equivalente a /produtos/, mas de forma dinâmica e desacoplada.

Templates do App Produtos Templates

Arquivos HTML dentro do app com namespace de pasta

Os templates do app ficam em produtos/templates/produtos/. A subpasta com o nome do app é um namespace de templates: garante que o arquivo seja identificado unicamente como produtos/lista.html e não entre em conflito com um hipotético lista.html de outro app.

produtos/templates/produtos/lista.html criar
🌐 produtos/templates/produtos/lista.html
<!DOCTYPE html> <html lang="pt-br"> <head> <meta charset="UTF-8"> <title>Produtos</title> </head> <body> <h1>Lista de Produtos</h1> <a href="{% url 'home' %}"> Voltar </a> </body> </html>
produtos/templates/produtos/detalhes.html criar
🌐 produtos/templates/produtos/detalhes.html
<!DOCTYPE html> <html lang="pt-br"> <head> <meta charset="UTF-8"> <title>Detalhes</title> </head> <body> <h1>Detalhes do Produto</h1> <a href="{% url 'produtos:lista' %}"> Voltar </a> </body> </html>
Os diretórios precisam ser criados manualmente. No terminal: mkdir -p produtos/templates/produtos

{% load static %} e Uso de CSS Static

Carregando a tag e vinculando folhas de estilo

Para usar arquivos estáticos em um template Django, é obrigatório carregar a biblioteca de tags estáticas com {% load static %} no início do template. Em seguida, usa-se a tag {% static 'caminho/do/arquivo' %} para gerar a URL correta do arquivo.

Criando o arquivo CSS do app criar
🎨 produtos/static/produtos/style.css
/* Estilos do app produtos */ body { font-family: Arial, sans-serif; background-color: #f5f5f5; padding: 20px; } h1 { color: #2c3e50; border-bottom: 2px solid #3498db; padding-bottom: 8px; }
Usando {% load static %} no template lista.html
🌐 produtos/templates/produtos/lista.html
<!-- 1. Sempre a primeira linha — carrega as template tags de estáticos --> {% load static %} <!DOCTYPE html> <html lang="pt-br"> <head> <meta charset="UTF-8"> <title>Produtos</title> <!-- 2. {% static %} gera a URL correta para o arquivo CSS --> <link rel="stylesheet" href="{% static 'produtos/style.css' %}"> </head> <body> <h1>Lista de Produtos</h1> <a href="{% url 'home' %}">Voltar</a> </body> </html>
{% load static %} deve estar presente em cada template que usa arquivos estáticos. Não é herdado de outros templates. Esquecer essa linha causa o erro: Invalid block tag: 'static'.

Uso de Imagens e JavaScript Static

Aplicando {% static %} para todos os tipos de arquivo

A tag {% static %} funciona da mesma forma para qualquer tipo de arquivo estático: imagens, scripts JS, fontes, ícones, etc. O caminho passado é relativo à raiz de um dos diretórios de estáticos configurados (STATICFILES_DIRS ou static/ do app).

Estrutura dos arquivos a criar
produtos/static/produtos/ ├── style.css ├── script.js └── img/ └── banner.png
produtos/static/produtos/script.js criar
📜 produtos/static/produtos/script.js
// Script simples de exemplo console.log('App produtos carregado!');
Template com imagem e JS editar
🌐 produtos/templates/produtos/lista.html
{% load static %} <!DOCTYPE html> <html lang="pt-br"> <head> <meta charset="UTF-8"> <title>Produtos</title> <link rel="stylesheet" href="{% static 'produtos/style.css' %}"> </head> <body> <h1>Lista de Produtos</h1> <!-- Exibindo uma imagem estática --> <img src="{% static 'produtos/img/banner.png' %}" alt="Banner Produtos" width="300"> <a href="{% url 'home' %}">Voltar</a> <!-- Script JS ao final do body --> <script src="{% static 'produtos/script.js' %}"></script> </body> </html>
